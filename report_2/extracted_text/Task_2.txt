Cloud-based Data Warehouse Architecture  -- Amazon Redshift
https://docs.aws.amazon.com/redshift/latest/dg/c_high_level_system_architecture.html 
Amazon Redshift is a relational database management system (RDBMS)
Amazon Redshift is based on PostgreSQL 8.0.2 and is somewhat different
The leader node manages communications with client programs and all communication with compute nodes.
Each compute node has its own dedicated CPU, memory, and attached disk storage, which are determined by the node type.
A compute node is partitioned into slices.
User data is stored on the compute nodes. Your SQL client communicates with the leader node, which in turn coordinates query execution with the compute nodes.

4. Focusing on Technical Architecture 

We divide the entire technical architecture into two big pieces: back room and front room. 
 
There are two major types of components in the technical architecture: services and data stores.  Services are functions needed to accomplish the required tasks of the warehouse.  For example, copying tables from one to another is a data transport service.  Data stores are temporary or permanent landing places for data.
 
 
Key technical issues:
Metadata
Flexible service layers with the help of metadata catalog

Service 1:  Extraction Services
60% of the DW development hours are spent on the extract process.
Most often, the challenge is determining what data to extract and what kinds of filters to apply.
Most extract processes generate temporary load files that become the input data to the next activity downstream.
Most extracts are based on the host platform and are designed to get the data out of the source and into a simplified and easily accessible form.
Data needs to be pulled from the multiple systems built with multiple data stores hosted on multiple platforms ...

  Two kinds of Presentation Servers:
  Enterprise-wide Data Warehouse (Inmon)
  Data warehouse is a collection of data marts that sit on a bus  (Kimball)

Atomic data marts
it holds data at the lowest-common-denominator level i.e., lowest level of detail necessary to meet most of the high-value business requirements.
Atomic data marts should be relational due to the details in the data.  
Atomic data marts should be built around dimensional model with details.

Selection of Target DBMS for DW/DM
Choice 1:	Conventional RDBMSs in combination with ROLAP tools
Strengths:
No size limit on target database (scalable to terabytes)
No need for proprietary databases and tools
RDBMSs have been enhanced to support the specialized requirements for DW.  Techniques include parallel query execution, parallel data management, bit-mapped indexes, SQL extensions for OLAP, etc.
Ad hoc queries can be directed to any data in the DW
Detailed data and aggregated data can be stored in the same database

.

Selection of Target DBMS for DW/DM
Recommendations 
Use RDBMS and ROLAP tools to provide multidimensional views of large relational target databases
Use RDBS features, such as parallelization, and bit-mapped indexes, to provide acceptable performance
Use MDB for high-performance analysis of moderate size databases.  MDBs are often used for data marts.
Use hybrid approach for applications that require access to detail data and fast OLAP computations.
