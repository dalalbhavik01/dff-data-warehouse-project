Step 2.  Translated Relations are as follows:
Employee (enum, employee-name, employee-addr)
Supplier (s-ID, s-name, s-address, enum)

Translation Rule 1
[a] Each entity type in an ERD (that is not ”weak”) is transformed to a relation.
[b] The PK of the entity becomes the PK of the relation
[c] The non-key attributes of the entity become the non-key attributes of the relation. 
Translation Rule 2
[a]For each weak entity type generate a new relation.
[b] The PK is the composite key created by concatenation of its own key and primary key of the supporting entity.
[c] The non-key attributes of the entity become the non-key attributes of the relation. 
Translation Rule 3
For each n-ary relationship, generate a corresponding relation.  The key is the “composite key” consisting of primary key for each of the entities in the relationship. 
Translation Rule 4
For each binary relationship WITH attributes, generate a new relation.  The key of this relation is the “composite key” consisting of primary key for each of the entities in the relationship.
Translation Rule 5
For each binary relationship WITHOUT attributes, generate a new relation only if the relationship is (many, many).  The key of this relation is the composite key consisting of primary key for each of the two entities in the relationship.
Translation Rule 6
For each binary relationship WITHOUT attributes, if the relationship is (1,many) or (1,1), place a foreign key in the relation that represents the entity on the many-side of the relationship.  The foreign key is the primary-key of the entity on one-side of the relationship.  For (1,1), the foreign key can be placed in either relation.

I.  ERD and Relational Schema
Step 3
First normal form (1NF) is a property of a relation in a relational database. A relation is in first normal form if and only if the domain of each attribute contains only atomic (indivisible) values, and the value of each attribute contains only a single value from that domain.
A relation is in the second normal (2NF) form if it fulfills the following two requirements: 
It is in first normal form.
It does not have any non-prime attribute that is functionally dependent on any proper subset of any candidate key of the relation. A non-prime attribute of a relation is an attribute that is not a part of any candidate key of the relation.
A database relation is said to meet third normal form (3NF) standards if all the attributes (e.g. database columns) are functionally dependent on solely the primary key. Codd defined this as a relation in second normal form where all non-prime attributes depend only on the candidate keys and do not have a transitive dependency on another key.

 7.   Why Need Multiple Fact Tables?
  Notion of Aggregates
  Aggregates are stored summaries built primarily to improve query performance.
  The creation of aggregates introduces a family of fact tables that are usually derived from the most granular fact table in each data mart. 
  Each member of the family represents a particular degree of summarization alone one or more of the fact table’s dimensions.
  A mature and full set of aggregates could involve a dozen separate aggregate fact tables, all echoing the original structure of the base fact table.

Snowflake Schema
12.  Permissible Snowflaking
   A dimension is said to be “snowflaked” when the low cardinality fields in the dimension have been removed to separate tables and linked back into the original table with artificial key.  This helps if the user is asking only for the distinct values of the category….
  Snowflaking makes most forms of browsing among the dimension attributes slower.
  Not always a good idea because --
    creates confusion with the users’ understanding
    create problems with “bit mapped indexes”
    more administration cost
(Low-cardinality refers to columns with few unique values. Low-cardinality column values are typically status flags, Boolean values, or major classifications such as gender. An example of a data table column with low-cardinality would be a CUSTOMER table with a column named NEW_CUSTOMER. This column would contain only 2 distinct values: Y or N, denoting whether the customer was new or not. Since there are only 2 possible values held in this column, its cardinality type would be referred to as low-cardinality. – Wikipedia)
